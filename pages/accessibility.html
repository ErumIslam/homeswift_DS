Accessibility

HomeSwift is committed to creating accessible experiences for all users, regardless of abilities. This section provides guidelines for ensuring that all HomeSwift interfaces are accessible and inclusive.

Accessibility Principles

HomeSwift follows the Web Content Accessibility Guidelines (WCAG) 2.1 at the AA level. These guidelines are organized around four principles:

  • Perceivable: Information and user interface components must be presentable to users in ways they can perceive.
  • Operable: User interface components and navigation must be operable.
  • Understandable: Information and the operation of the user interface must be understandable.
  • Robust: Content must be robust enough that it can be interpreted by a wide variety of user agents, including assistive technologies.

Color Contrast

Sufficient color contrast ensures that text and interactive elements are perceivable by users with low vision or color vision deficiencies.

Guidelines

  • Maintain a minimum contrast ratio of 4.5:1 for all text elements
  • Use higher contrast ratios (7:1) for critical information
  • Avoid using color alone to convey information
  • Provide alternative visual cues alongside color coding

Design Specifications

  • Primary text: Midnight Blue (#00305C) on White (#FFFFFF) - 10.7:1
  • Secondary text: Gray (#808080) on White (#FFFFFF) - 4.6:1
  • Accent text: Green Lime (#37C640) on White (#FFFFFF) - 2.5:1 (use only for large text or UI elements)
  • Error text: Red (#FF0000) on White (#FFFFFF) - 4:1

Technical Requirements

  • Implement automated contrast checking in design tools
  • Provide alternative text for color-coded elements
  • Test with various display settings (grayscale, high contrast)
  • Consider runtime contrast adjustment capabilities for user preferences

Contrast Checking Tool

function calculateContrastRatio(foreground, background) {
  // Convert hex to RGB
  const fgRGB = hexToRgb(foreground);
  const bgRGB = hexToRgb(background);
  
  // Calculate luminance
  const fgLuminance = calculateLuminance(fgRGB);
  const bgLuminance = calculateLuminance(bgRGB);
  
  // Calculate contrast ratio
  const ratio = (Math.max(fgLuminance, bgLuminance) + 0.05) / 
                (Math.min(fgLuminance, bgLuminance) + 0.05);
  
  return ratio.toFixed(1);
}

// Usage
const ratio = calculateContrastRatio('#00305C', '#FFFFFF');
console.log(`Contrast ratio: ${ratio}:1`);
// Output: "Contrast ratio: 10.7:1"

Text Readability

Readable text ensures that content is accessible to all users, including those with visual impairments or reading disabilities.

Guidelines

  • Use Poppins font throughout the application
  • Maintain minimum font sizes:
    • Body text: 16px
    • Captions: 14px
    • Headings: 24px+
  • Use appropriate line spacing (1.5x font size)
  • Limit line length to 80 characters for optimal readability

Text Readability Examples

This is body text at 16px with 1.5x line height. It provides optimal readability for most users and ensures that text is legible on all devices. The line length is limited to improve readability.

This is caption text at 14px with 1.4x line height. It's used for secondary information and should be limited in length.

Design Specifications

  • Font: Poppins throughout
  • Weight: Regular (400) for body, Medium (500) for headings
  • Letter spacing: 0.5px for body text
  • Line height: 24px for 16px body text

Technical Requirements

  • Implement text scaling support
  • Support dynamic type implementation
  • Consider localization requirements for text expansion
  • Test readability across different languages

Responsive Typography

html {

}

body {
  font-family: 'Poppins', sans-serif;

  font-size: 1rem; /* 16px at default size */
  line-height: 1.5;
  max-width: 80ch; /* Limit line length to 80 characters */

Navigation Accessibility

Accessible navigation ensures that all users can navigate the application efficiently regardless of abilities.

Guidelines

  • Implement logical tab order for keyboard navigation
  • Provide visible focus indicators for all interactive elements
  • Include skip navigation links for screen reader users
  • Ensure touch targets are at least 44x44px for mobile interfaces
  • Provide multiple navigation methods (search, menus, links)

Focus Indicators

Design Specifications

  • Focus state: 2px Green Lime (#37C640) outline with 2px offset
  • Touch targets: Minimum 44x44px with adequate spacing
  • Skip links: Visually hidden until focused, then appearing at top of viewport

Technical Requirements

  • Implement keyboard navigation testing
  • Ensure proper focus management
  • Verify touch target sizes
  • Test with screen readers for navigation accessibility

Screen Reader Support

Screen reader support ensures that content is accessible to users who rely on screen readers and other assistive technologies.

Guidelines

  • Use semantic HTML elements appropriately
  • Provide descriptive alt text for all images
  • Implement ARIA labels and landmarks where needed
  • Ensure form elements have associated labels
  • Announce dynamic content changes

Semantic HTML Example

<!-- Good: Using semantic HTML -->
<article>
  <h2>Service Details</h2>
  <p>This service includes professional cleaning of all rooms.</p>
  <figure>
    <img src="cleaning.jpg" alt="Professional cleaner wiping kitchen counter" width="300" height="200">
    <figcaption>Our professional cleaning service</figcaption>
  </figure>
</article>

<!-- Bad: Using non-semantic HTML -->
<div>
  <div>Service Details</div>
  <div>This service includes professional cleaning of all rooms.</div>
  <div>
    <img src="cleaning.jpg" width="300" height="200">
    <div>Our professional cleaning service</div>
  </div>
</div>

Design Specifications

  • Alt text guidelines: Concise, descriptive, under 125 characters
  • ARIA landmark roles: banner, navigation, main, complementary, contentinfo
  • Form labels: Visible, associated with inputs, descriptive

Technical Requirements

  • Test with screen readers across platforms
  • Verify ARIA implementation
  • Test dynamic content announcements
  • Validate form accessibility

ARIA Landmarks

<header role="banner">
  <!-- Site header content -->
</header>

<nav role="navigation" aria-label="Main Navigation">
  <!-- Navigation content -->
</nav>

<main role="main" id="main-content">
  <!-- Main content -->
</main>

<aside role="complementary" aria-label="Related Information">
  <!-- Sidebar content -->
</aside>

<footer role="contentinfo">
  <!-- Footer content -->
</footer>

Dynamic Content Announcements

<div aria-live="polite" class="visually-hidden">
  <!-- Content will be announced when changed -->
</div>

JavaScript for Announcements

function announceToScreenReader(message) {
  const announcer = document.querySelector('[aria-live="polite"]');
  announcer.textContent = message;
  
  // Clear after a delay to allow for multiple announcements
  setTimeout(() => {
    announcer.textContent = '';
  }, 3000);
}

// Usage
announceToScreenReader('Your booking has been confirmed.');

Accessibility Testing

Regular accessibility testing ensures that HomeSwift interfaces remain accessible as they evolve.

Testing Checklist

  • Automated Testing:
    • Use tools like Axe, WAVE, or Lighthouse
    • Integrate accessibility testing into CI/CD pipeline
    • Run automated tests on all major pages and components
  • Manual Testing:
    • Test keyboard navigation
    • Test with screen readers (NVDA, VoiceOver, JAWS)
    • Test with browser zoom (up to 200%)
    • Test with high contrast mode
    • Test with reduced motion settings
  • User Testing:
    • Include users with disabilities in user testing
    • Test with assistive technology users
    • Gather feedback on accessibility issues

Testing Tools

  • Automated Tools: Axe, WAVE, Lighthouse, Pa11y
  • Screen Readers: NVDA (Windows), VoiceOver (macOS/iOS), TalkBack (Android)
  • Contrast Checkers: WebAIM Contrast Checker, Colour Contrast Analyser
  • Keyboard Testing: Manual keyboard navigation, Keyboard Navigation Viewer

Testing Process

  1. Run automated tests to identify basic issues
  2. Conduct manual testing to identify more complex issues
  3. Prioritize issues based on impact and frequency
  4. Fix issues and retest
  5. Document accessibility features and limitations
  6. Conduct regular accessibility audits
